# ERR-0036: Cascade deletion destroys chat segment analysis

## What was broken (symptoms)
When running the `replace_chat_segments` function during subsequent full passes of chat synchronization, all old segments for that chat were unconditionally deleted and re-inserted. Since `chat_segment_analysis` maps to `chat_segments(segment_id)` with an `ON DELETE CASCADE` constraint, the expensive LLM analysis results were implicitly destroyed immediately upon chat re-analysis.

## Investigation notes (facts, logs, repro steps)
The `replace_chat_segments` codebase logic unconditionally ran:
`DELETE FROM chat_segments WHERE chat_id = $1`
before inserting new objects generated by the segmenter. Because the LLM results are directly attached to these DB IDs, this wiped out any previously aggregated results whenever a chat was refreshed.

## Hypotheses considered (and rejected)
- Remove `ON DELETE CASCADE`: Rejected, because orphaned rows are bad constraints and we actually DO want them deleted if a chat completely vanishes or shrinks drastically.
- Use `ON CONFLICT` Upsert: Accepted.

## Experiments tried (what worked / what didn’t)
- Refactoring `replace_chat_segments` to use `ON CONFLICT (chat_id, segment_no) DO UPDATE` so that identical segments just get updated (keeping their autoincrement ID intact), and preserving the downstream LLM processing cache.
- However, since there was no UNIQUE constraint on `(chat_id, segment_no)`, the UPSERT command failed. Added a new Piccolo Auto-Migration + raw query to add `uq_chat_segments_chat_id_segment_no`.

## Final fix + why it works
- Added `class Meta(): unique_together = (("chat_id", "segment_no"),)` to `etl/tables.py`.
- Created a Piccolo migration to apply `ALTER TABLE chat_segments ADD CONSTRAINT uq_chat_segments_chat_id_segment_no UNIQUE (chat_id, segment_no);`.
- Upgraded the DB Python operation to an `INSERT ... ON CONFLICT (...) DO UPDATE` statement, capturing all the existing and new IDs.
- Purged leftovers with `DELETE FROM chat_segments WHERE chat_id = $1 AND id != ALL($2::int[])` at the end to clean up if the new segments length falls short of the old.

## Verification (tests, commands, expected output)
- `docker exec summagram_new-postgres-1 psql -U summagram -d summagram -c "\d+ chat_segments"` verified the unique constraint was added.
