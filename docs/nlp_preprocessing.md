# NLP Preprocessing Pipeline

В рамках архитектуры Summoner/Summagram реализован многоступенчатый детерминированный NLP пайплайн (Session Pipeline) для предварительной обработки запросов пользователей в сессиях (`backend.session_pipeline`). Этот слой работает **перед** основной LLM и снимает с нее задачи извлечения именованных сущностей (NER) и парсинга времени.

## Зачем это нужно?

Изначально вся нагрузка по определению того, что именно пользователь хочет (Intent, Время, Контакт), ложилась на плечи LLM (Qwen, LLaMA и т.д.) через огромный `SESSION_INTENT_PROMPT`. 
Для морфологически богатых языков, таких как русский (RU) или украинский (UK), LLM часто ошибалась:
* **Проблемы с падежами** ("Что писала Алиса?" $\rightarrow$ LLM часто не приводила к именительному падежу "Алиса", а пыталась искать "Алисой" и сливать имя в `search_query`).
* **Ошибки формата дат**: Относительные даты ("позавчера", "две недели назад") было сложно конвертировать в точные timestamp для фильтров базы данных локальной LLM.
* **Слишком тяжелые промпты**: Инструкции по извлечению дат и правил парсинга контактов "засоряли" контекст.

**Решение**: Использовать классические детерминированные инструменты NLP (regex, синтаксический анализ, словари), чтобы извлечь контакты, время и URL, а LLM использовать исключительно для смыслового связывания.

## Архитектура 5 Стадий

Пайплайн разбит на 5 изолированных стадий (определены в `interfaces/protocols.py`). Главный оркестратор — `SessionPipeline` (в `services/pipeline.py`).

### Stage 0. Normalization (Нормализация)
**Инструмент**: Набор Regex-правил (`DefaultTextNormalizer`).
*   **Очистка**: Удаляет невидимые Unicode-символы и лишние пробелы.
*   **Извлечение ссылок/ников**: Изымает `@usernames` и `URLs`, чтобы они не смущали следующие ML-шаги (например, Natasha NER может оботкнуть об URL).
*   **Language Hint**: Делает простую эвристику (наличие специфичных укр. букв `і, ї, є` или рус. `ы, э, ъ`), чтобы выставить `LanguageCode.RU`, `UK`, `EN` или `MIXED`.

### Stage 1. Candidate Extraction (Извлечение Кандидатов)
**Инструмент**: Композитный механизм (`CompositeCandidateExtractor`).
*   **Rule-based (`RulesCandidateExtractor`)**: Извлекает кандидатов напрямую. Если человек написал "от @john_doe", "що задала Аліса" или взял слово в кавычки, мы извлекаем это как `CandidateMention`.
*   **ML-based (`NatashaCandidateExtractor`)**: Для текстов, размеченных как RU/MIXED, запускается библиотека `natasha`. Она использует Slovnet для распознавания токенов типа `PER` (Person). Обернуто в `asyncio.to_thread`, чтобы не блокировать event loop.
*   *Опционально*: Модель `lang-uk` через HuggingFace `transformers` (будет использоваться для UK текстов).

### Stage 2. Entity Resolution (Разрешение Сущностей)
**Инструмент**: Морфологический анализ + Запросы в БД (`DefaultEntityResolver`).
*   Получает список сущностей (`CandidateMention`).
*   **Приведение к Именительному Падежу (Lemmatization)**: Библиотека `pymorphy3` переводит ФИО из любого падежа (например, "Алисой") в именительный ("Алиса"). 
*   **Построение Вариантов**: Создает латинские/кириллические варианты и приставки (`build_multilingual_prefix_patterns`).
*   **Database Lookup**: Проверяет через базу данных `etl.db.chats.find_chats_by_contact_name_sync`, есть ли такой реальный контакт.
*   **Fuzzy Matching**: Если жесткое сопоставление префиксов не сработало, применяется `RapidFuzz` для поиска похожих имен в кэше профилей. Порог задан как 80% совпадения.

### Stage 3. Time Parsing (Парсинг Времени)
**Инструмент**: Библиотека `dateparser` (`DateparserTimeParser`).
*   Конвертирует фразы вроде "за останні 2 дні", "yesterday", "с понедельника" в конкретный `ParsedTimeExpression` (даты от и до).
*   Логика включает корректировку маппинга для русского и английского словаря (поддержка `_UNIT_MAP`).

### Stage 4. Intent Classification (Классификация намерений LLM)
**Инструмент**: `generate_json` через `sglang` клиент (`LlmIntentClassifier`).
*   Только после того, как сущности и время найдены, остаток текста отправляется в LLM.
*   LLM смотрит на остаток запроса ("отчёты") и на список уже найденных entities.
*   Возвращает структурированный `QueryIntent`: тип запроса (`query_type` – `search_text`, `summarize`, и т.д.), ключевые слова для поиска (`search_query` без имени человека) и лимиты.

## Интеграция в основной процесс (Session Agent)

Весь процесс запускается в главном рут-агенте (`backend/session_agent.py`):

1. При старте обработки сообщения вызывается асинхронный адаптер:
   ```python
   result = await _run_nlp_pipeline(session_id, context_chat_id, message, chat_history)
   ```
2. Анализируется `result.entities.primary_person` (если он есть).
3. Анализируется `result.time_range`.
4. Из `result.intent` извлекаются `query_type` и `search_query`.
5. Питон вызывает синхронные инструменты агрегации: `find_chat_by_contact_name_sync`, `get_last_messages_sync`.
6. Собранный контекст отправляется в `SESSION_SYNTHESIS_PROMPT` через `sglang_client.generate_text`.
7. Финальный ответ отдается пользователю.

### Флаг Фичи (Feature Flag)
Для гибкости, пайплайн включается опцией в `shared/config.py`:
```env
NLP_PIPELINE_ENABLED=true
```
Если флаг = `false`, система использует старый механизм, где LLM должна самостоятельно "вспоминать" `person_name` и даты.
